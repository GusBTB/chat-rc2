ETAPA 1 - Criacao do banco SQLite

Nesta etapa foi criada a estrutura inicial do banco de dados SQLite do projeto.

O arquivo do banco fica localizado em:

database/chat.db

Foi criada a classe DatabaseConnection, responsavel por fornecer conexao com o banco SQLite usando JDBC.

Tambem foi criada a classe DatabaseInitializer, responsavel por criar automaticamente as tabelas necessarias do sistema quando a aplicacao e iniciada.

Tabelas criadas:

1. users
   Guarda os dados dos usuarios cadastrados, incluindo nome completo, login, email, senha e status.

2. blocks
   Guarda os bloqueios entre usuarios.

3. direct_permissions
   Guarda as permissoes de conversa privada entre usuarios.

4. groups_chat
   Guarda os grupos criados no chat.

5. group_members
   Guarda os membros dos grupos e informa se cada membro e administrador.

6. messages
   Guarda as mensagens enviadas no sistema.

7. message_deliveries
   Guarda o controle de entrega das mensagens, permitindo identificar mensagens entregues e mensagens pendentes.

8. pending_requests
   Guarda convites e solicitacoes pendentes, como convite para grupo, pedido de conversa privada, solicitacao de entrada em grupo e convite para ser administrador.

Ao executar a classe App, o sistema inicializa o banco e cria as tabelas automaticamente caso elas ainda nao existam.

ETAPA 2 - Criacao dos modelos Java

Nesta etapa foram criadas as classes que representam os principais dados do sistema.

Foi criado o pacote:

br.edu.chat.model

Classes e enums criados:

1. UserStatus
   Enum que representa os status possiveis de um usuario:
   ONLINE, OFFLINE, OCUPADO e AUSENTE.

2. MessageType
   Enum que representa os tipos de mensagens do sistema:
   DIRECT, GROUP, GROUP_DIRECT, SYSTEM, INVITE e REQUEST.

3. User
   Classe que representa um usuario do chat.
   Possui id, nome completo, login, email, senha e status.

4. Group
   Classe que representa um grupo do chat.
   Possui id, nome do grupo e id do usuario criador.

5. Message
   Classe que representa uma mensagem.
   Possui id, usuario remetente, usuario destinatario, grupo, conteudo, tipo da mensagem e data de criacao.

As classes de modelo nao possuem dependencia de socket, console ou banco de dados.
Elas apenas representam dados usados pelo sistema.

Tambem foi criada a classe temporaria ModelTest, usada para testar a criacao de objetos User, Group e Message.

Foi criada a classe temporaria DatabaseUserTest, usada para inserir e consultar um usuario no banco SQLite usando JDBC diretamente.
Essa classe confirmou que o banco chat.db esta funcionando corretamente.
Posteriormente essa logica sera movida para a camada de repositorios.

ETAPA 3 - Camada de persistencia

Tambem foi criada a classe MessageDelivery, que representa uma mensagem pendente de entrega para um usuario.

Foi criada a classe temporaria RepositoryTestMain para testar a camada de repositorios.

Testes realizados:

ETAPA 4 - Protocolo textual e parser de comandos

Nesta etapa foi criada a camada de protocolo do sistema, responsavel por interpretar as linhas de texto digitadas pelo usuario e transformar essas linhas em objetos de comando que poderao ser processados pelo nucleo da aplicacao.

Foi criado o pacote:

br.edu.chat.protocol

Classes e enums criados:

1. CommandType
   Enum que representa todos os tipos de comandos aceitos pelo sistema:
   CADASTRO, LOGIN, LOGOUT, RECUPERAR_SENHA, STATUS,
   LISTA_USUARIOS, LISTA_GRUPOS, LISTAR_GRUPO,
   DIRECT_MESSAGE, GROUP_MESSAGE, GROUP_DIRECT_MESSAGE,
   NOVO_GRUPO, INSERIR_GRUPO, ENTRAR_GRUPO, SAIR_GRUPO, PROMOVER_ADMIN,
   BLOQUEAR, DESBLOQUEAR,
   ACEITAR, RECUSAR,
   AJUDA, INVALID, UNKNOWN.

2. ClientCommand
   Classe imutavel que representa um comando ja interpretado.
   Possui o tipo do comando, a linha original, uma lista de argumentos,
   uma lista de usuarios alvo, o nome do grupo, o conteudo da mensagem,
   o id de pedido (para aceitar/recusar) e uma mensagem de erro quando aplicavel.
   A construcao e feita por meio de Builder interno.

3. CommandParser
   Classe responsavel por receber uma linha de texto e retornar um ClientCommand.
   Reconhece os formatos descritos no enunciado do trabalho:
   - cadastro;Nome;login;email;senha
   - login;login;senha
   - logout
   - recuperarsenha;email
   - status;ONLINE | OFFLINE | OCUPADO | AUSENTE
   - listausuarios
   - listagrupos
   - listargrupo &nomegrupo
   - novogrupo nomegrupo
   - inserir &nomegrupo@usuario1,@usuario2
   - entrar &nomegrupo
   - sair &nomegrupo
   - promover &nomegrupo@usuario
   - bloquear @usuario
   - desbloquear @usuario
   - aceitar id
   - recusar id
   - @usuario: mensagem
   - &grupo: mensagem
   - &grupo@usuario1,@usuario2: mensagem
   - ajuda
   Comandos invalidos retornam um ClientCommand do tipo INVALID, com mensagem de erro amigavel.
   Comandos nao reconhecidos retornam um ClientCommand do tipo UNKNOWN.

4. ServerResponse
   Classe utilitaria com prefixos e formatadores para mensagens enviadas pelo servidor.
   Define prefixos padronizados como [OK], [ERRO], [INFO], [SISTEMA], [CONVITE id], [PEDIDO id] e [SOLICITACAO id].
   Possui metodos para formatar mensagens diretas, mensagens de grupo e mensagens diretas dentro de grupo.

Foi criada a classe temporaria CommandParserTest, com 44 verificacoes que cobrem cada formato de comando, casos com erro (campos vazios, formato incorreto, status invalido, id nao numerico, comando vazio ou nulo) e os casos UNKNOWN.

Testes realizados (todos passaram):
- reconhecimento de cadastro, login, logout, recuperarsenha e status;
- reconhecimento de listausuarios, listagrupos e listargrupo;
- reconhecimento de mensagem direta @usuario: mensagem;
- reconhecimento de novogrupo, inserir (um e varios usuarios), entrar, sair e promover;
- reconhecimento de mensagem de grupo &grupo: mensagem;
- reconhecimento de mensagem direta em grupo &grupo@usuario: mensagem;
- reconhecimento de mensagem direta em grupo para varios &grupo@u1,@u2: mensagem;
- reconhecimento de bloquear e desbloquear;
- reconhecimento de aceitar e recusar com id numerico;
- retorno INVALID para campos faltando, formato errado, mensagem vazia, status invalido, id nao numerico;
- retorno UNKNOWN para comando inexistente;
- retorno INVALID para comando vazio e para entrada nula.

Esta camada nao depende de socket nem de banco de dados.
Ela apenas converte texto em objetos de comando, deixando o nucleo da aplicacao livre para tomar decisoes sobre cada comando recebido.

ETAPA 5 - Criacao e correcao do nucleo de servicos

Nesta etapa foi criada a camada de servicos do sistema, localizada no pacote:

br.edu.chat.core

A camada de servicos concentra as regras de negocio do chat. Ela fica entre o protocolo textual/servidor e a camada de repositorios. Dessa forma, o codigo de socket nao precisa conhecer diretamente os detalhes do banco de dados nem as regras completas de usuario, grupo, mensagens e bloqueios.

Arquivos principais criados nesta etapa:

1. UserService
   Responsavel pelas regras de usuario:
   - cadastro;
   - login;
   - logout;
   - recuperacao de senha;
   - listagem de usuarios;
   - listagem de usuarios online;
   - entrega de mensagens pendentes.

2. MessageService
   Responsavel pelas regras de mensagens:
   - envio de mensagem direta;
   - envio de mensagem para grupo;
   - envio de mensagem privada dentro de grupo;
   - salvamento de mensagens pendentes;
   - verificacao de bloqueios;
   - confirmacao de entrega ao remetente.

3. GroupService
   Responsavel pelas regras de grupo:
   - criacao de grupo;
   - convite de usuario para grupo;
   - pedido de entrada em grupo;
   - saida de grupo;
   - convite para promover administrador;
   - listagem de grupos;
   - listagem de membros do grupo.

4. BlockService
   Responsavel pelas regras de bloqueio:
   - bloquear usuario;
   - desbloquear usuario;
   - verificar se existe bloqueio entre dois usuarios.

5. RequestService
   Responsavel por processar pedidos pendentes:
   - aceitar pedido de mensagem direta;
   - recusar pedido de mensagem direta;
   - aceitar convite para grupo;
   - recusar convite para grupo;
   - aceitar pedido de entrada em grupo;
   - recusar pedido de entrada em grupo;
   - aceitar convite para se tornar administrador;
   - recusar convite para se tornar administrador.

6. MessageSender
   Interface criada para separar a regra de negocio do envio real de mensagens.
   Nesta etapa, os servicos ainda nao dependem diretamente de socket.
   O envio real sera conectado ao servidor TCP em uma etapa posterior.

O que ja havia sido implementado:

- Cadastro de usuario.
- Login e logout.
- Listagem de usuarios.
- Listagem de usuarios online.
- Criacao de grupos.
- Convite para grupo.
- Pedido de entrada em grupo.
- Envio de mensagens diretas.
- Envio de mensagens para grupo.
- Envio de mensagens privadas dentro de grupo.
- Bloqueio e desbloqueio de usuarios.
- Pedidos pendentes com aceitar e recusar.
- Estrutura inicial para mensagens offline.

Correcoes realizadas nesta etapa:

1. Correcao da permissao de mensagem direta
   Antes, ao enviar a primeira mensagem direta, o sistema criava o pedido de permissao e tambem liberava a permissao automaticamente.
   Isso estava incorreto, pois o destinatario deve aceitar antes da conversa direta ser liberada.
   Agora, o sistema apenas cria o pedido pendente.
   A permissao so e concedida quando o destinatario usa o comando aceitar.

2. Correcao do controle de pedidos pendentes
   Foi ajustado o uso de pending_requests para controlar pedidos ainda nao respondidos.
   Isso evita criar permissoes antes da hora e tambem evita pedidos duplicados em alguns fluxos.

3. Correcao do envio de mensagem em grupo
   Agora as mensagens de grupo tambem respeitam bloqueios entre usuarios.
   Se A bloqueou B ou B bloqueou A, mensagens de grupo de um nao sao entregues ao outro.

4. Correcao do envio de mensagem privada dentro do grupo
   O envio privado dentro do grupo verifica se o destinatario existe, se pertence ao grupo e se nao existe bloqueio entre remetente e destinatario.

5. Correcao do pedido de entrada em grupo
   Antes, o pedido de entrada era enviado apenas ao criador ou administrador principal do grupo.
   Agora, o pedido e enviado para todos os membros atuais do grupo.
   O usuario solicitante so entra no grupo quando todos os membros aceitarem.
   Se qualquer membro recusar, a entrada e negada.

6. Correcao da promocao de administrador
   Antes, o usuario era promovido diretamente a administrador.
   Agora, o administrador envia um convite de promocao.
   O usuario convidado precisa aceitar para se tornar administrador.

7. Correcao da saida de grupo
   Agora, se o usuario for o unico administrador do grupo, ele nao pode sair antes de promover outro usuario a administrador.
   Essa regra evita que um grupo fique sem administrador.

8. Melhoria nas respostas de envio
   As respostas do sistema passaram a informar melhor:
   - quantos usuarios receberam a mensagem imediatamente;
   - quantos receberao depois;
   - quantos nao receberam por bloqueio ou por serem invalidos.

Observacao sobre mensagens pendentes:

O metodo UserService.deliverPendingMessages(User user) ja existe e sera usado para entregar mensagens pendentes quando o usuario ficar online.

A chamada desse metodo deve ser feita na etapa do servidor TCP, depois que:
1. o usuario fizer login com sucesso;
2. o ClientHandler for associado ao usuario;
3. o usuario for registrado no ClientSessionManager.

Isso evita tentar entregar mensagens antes de o usuario ter uma conexao TCP ativa.

Estado final da Etapa 5:

A camada de servicos esta preparada para ser integrada ao servidor TCP.
A regra de negocio principal esta separada da camada de socket.
As proximas etapas devem criar o servidor TCP, o ClientHandler, o ClientSessionManager e o CommandProcessor para ligar os comandos digitados pelos clientes aos servicos criados nesta etapa.
Complemento de validacao da Etapa 5

Foram realizados testes manuais adicionais no ServiceTestMain para validar regras importantes do nucleo de servicos.

Resultados validados:

1. Promocao de administrador com aceite
   Foi validado que um usuario nao se torna administrador imediatamente quando recebe o convite.
   Primeiro, o convite de administracao fica pendente.
   Somente depois que o usuario convidado aceita o pedido, ele passa a aparecer como administrador do grupo.

2. Pedido de entrada em grupo com mais de um membro aprovador
   Foi validado que, quando um usuario solicita entrada em um grupo que possui mais de um membro, todos os membros atuais recebem uma solicitacao.
   O usuario solicitante nao entra no grupo apos apenas uma aprovacao.
   Ele so entra depois que todos os membros aprovarem.

3. Bloqueio em mensagem de grupo
   Foi validado que o bloqueio tambem afeta mensagens enviadas dentro de grupos.
   Quando Bruno bloqueou Alice e Alice enviou uma mensagem para o grupo, Bruno nao recebeu a mensagem, mas os demais membros nao bloqueados receberam.
   A resposta do envio tambem indicou que houve usuario bloqueado/nao entregue.

Pendencia encontrada:

1. Mensagem offline no teste manual
   O teste manual mostrou que, apos o logout de Bruno, o MessageService ainda considerou Bruno online e entregou a mensagem imediatamente.
   Isso ocorreu porque o stubSender usado no ServiceTestMain nao estava sincronizado com o logout do UserService.
   O UserService alterava o status do usuario no banco, mas o stubSender continuava respondendo que Bruno estava online.

Correcao aplicada no teste manual:

Foi ajustado o stubSender do ServiceTestMain para manter uma lista de usuarios online.
Quando um usuario faz login, ele e adicionado a essa lista.
Quando um usuario faz logout, ele e removido dessa lista.
Assim, o teste manual consegue simular corretamente usuarios online e offline antes da integracao com sockets.

Observacao para etapa futura:

Na etapa do servidor TCP, esse controle de usuarios online nao devera depender do stubSender.
Ele sera feito pelo ClientSessionManager, que mantera os usuarios atualmente conectados ao servidor.
Ao fazer login, o usuario sera registrado no ClientSessionManager.
Ao fazer logout ou desconectar, o usuario sera removido.
Depois disso, a entrega de mensagens pendentes devera ser chamada com o usuario ja associado a uma conexao ativa.

ETAPA 6 - Criacao do servidor TCP

Nesta etapa foi criada a camada de servidor TCP do projeto.

O objetivo desta etapa foi permitir que clientes externos se conectem ao sistema por meio de sockets TCP. A partir desta etapa, o sistema deixa de depender apenas de testes manuais e passa a ter uma estrutura real de servidor, capaz de aceitar conexoes e criar uma thread para cada cliente conectado.

Arquivos criados:

1. br.edu.chat.server.ChatServer
   Classe responsavel por iniciar o ServerSocket na porta configurada.
   A thread principal do servidor fica aguardando conexoes usando serverSocket.accept().
   Quando um cliente se conecta, o servidor cria um ClientHandler e inicia uma nova Thread para atender esse cliente.

2. br.edu.chat.server.ClientHandler
   Classe que representa um cliente conectado ao servidor.
   Cada ClientHandler roda em uma thread propria.
   Suas responsabilidades sao:
   - ler linhas enviadas pelo cliente;
   - enviar essas linhas para o CommandProcessor;
   - enviar as respostas de volta ao cliente;
   - guardar qual usuario esta logado naquele socket;
   - remover a sessao do usuario quando o cliente desconectar.

3. br.edu.chat.server.ClientSessionManager
   Classe responsavel por controlar os usuarios online no servidor.
   Internamente utiliza ConcurrentHashMap para permitir acesso seguro por varias threads.
   Tambem implementa a interface MessageSender.
   Com isso, os servicos do nucleo conseguem enviar mensagens para usuarios online sem depender diretamente de sockets.

4. br.edu.chat.core.CommandProcessor
   Classe responsavel por conectar o protocolo textual com os servicos do nucleo.
   Ela recebe uma linha digitada pelo cliente, usa o CommandParser para interpretar o comando e chama o service adequado.
   Exemplos:
   - cadastro chama UserService.register;
   - login chama UserService.login;
   - @usuario: mensagem chama MessageService.sendDirect;
   - &grupo: mensagem chama MessageService.sendGroup;
   - novogrupo chama GroupService.createGroup;
   - aceitar id chama RequestService.accept;
   - bloquear chama BlockService.block.

5. br.edu.chat.MainServer
   Classe principal para iniciar o servidor TCP.
   Por padrao, usa a porta 6789.
   Tambem permite informar a porta por argumento.

Alteracao realizada:

O arquivo App.java foi ajustado para chamar MainServer.main(args), facilitando a execucao do servidor diretamente pela IDE.

Funcionamento do servidor:

1. O servidor inicializa o banco de dados.
2. O servidor abre um ServerSocket na porta definida.
3. A thread principal fica bloqueada em accept(), aguardando clientes.
4. Quando um cliente conecta, o servidor cria um ClientHandler.
5. Cada ClientHandler roda em uma thread separada.
6. O ClientHandler le comandos enviados pelo cliente.
7. O CommandProcessor interpreta e executa os comandos.
8. As respostas sao enviadas de volta ao cliente pelo socket.

Controle de usuarios online:

Nesta etapa, o controle de usuarios online passou a ser feito pelo ClientSessionManager.
Quando um usuario faz login com sucesso:
- o ClientHandler guarda o id e o login do usuario;
- o ClientSessionManager registra o usuario como online;
- as mensagens pendentes sao entregues apos o registro da sessao.

Quando um usuario faz logout ou desconecta:
- o usuario e removido do ClientSessionManager;
- o status no banco e atualizado para OFFLINE;
- o ClientHandler limpa os dados do usuario logado.

Status de usuario:

Quando o usuario altera o status para ONLINE, ele volta a ser registrado no ClientSessionManager e pode receber mensagens em tempo real.
Quando altera para OFFLINE, OCUPADO ou AUSENTE, ele continua conectado ao socket, mas sai da lista de usuarios disponiveis para entrega imediata.

Observacao:

Nesta etapa ainda nao foi criado o cliente Java de console.
Os testes podem ser feitos temporariamente usando telnet ou outro cliente TCP simples.
O cliente Java proprio sera criado na proxima etapa.

Testes previstos:

- iniciar servidor;
- conectar um cliente via telnet;
- verificar mensagem de boas-vindas;
- executar comando ajuda;
- cadastrar usuario via socket;
- fazer login via socket;
- conectar dois clientes ao mesmo tempo;
- enviar comandos sem derrubar o servidor;
- fechar um cliente e verificar que o servidor continua rodando;
- confirmar que usuario logado fica OFFLINE ao desconectar.

Complemento de validacao e ajustes da Etapa 6

Durante os primeiros testes manuais do servidor TCP, foram encontrados alguns pontos de ajuste.

1. Ajuste de quebras de linha nas respostas do servidor

Foi observado que respostas com varias linhas, como ajuda e listausuarios, apareciam desalinhadas em alguns clientes TCP.
Isso acontecia porque a resposta era enviada como uma unica string contendo quebras de linha internas.

Correcao aplicada:
O metodo send do ClientHandler foi ajustado para dividir mensagens com multiplas linhas e enviar cada linha separadamente pelo PrintWriter.
Com isso, respostas como ajuda, listausuarios, listagrupos e listargrupo passaram a aparecer com um item embaixo do outro.

2. Tratamento de caracteres de controle e backspace

Durante testes manuais, alguns comandos digitados com erro e corrigidos com Backspace eram interpretados como comandos desconhecidos.
Isso ocorre porque alguns clientes TCP enviam caracteres de controle, como Backspace ou DEL, dentro da linha recebida pelo servidor.

Correcao aplicada:
Foi criado um tratamento no ClientHandler para limpar a linha recebida antes de enviar ao CommandProcessor.
Esse tratamento remove caracteres de controle e aplica o efeito de Backspace sobre a string final.

3. Sugestao para comandos desconhecidos

Foi adicionada uma melhoria no CommandProcessor para sugerir comandos parecidos quando o usuario digita um comando desconhecido.
Exemplo:
Se o usuario digitar aceitr 1, o servidor pode sugerir aceitar.

Essa melhoria apenas sugere o comando.
O servidor nao executa automaticamente comandos corrigidos, pois comandos como aceitar e recusar alteram o estado do sistema e precisam ser confirmados explicitamente pelo usuario.

Pendencia futura identificada:

1. Primeira mensagem direta apos aceite

Durante o teste de mensagem direta, foi observado que, quando Alice envia uma primeira mensagem para Bruno sem permissao previa, o sistema cria corretamente o pedido de permissao.
Porem, apos Bruno aceitar, a mensagem original nao e entregue automaticamente.
Alice precisa enviar a mensagem novamente.

Esse comportamento esta coerente com a implementacao atual, porque o pedido de permissao nao armazena o conteudo da primeira mensagem.
Entretanto, pelo enunciado do trabalho, pode ser necessario que a primeira mensagem seja entregue apos o aceite.

Correcao futura recomendada para a etapa 10:
Ajustar a modelagem dos pedidos de mensagem direta para armazenar a primeira mensagem enquanto o pedido estiver pendente.
Uma alternativa e associar um message_id ao pedido pendente ou criar uma estrutura especifica para guardar a mensagem inicial ate o aceite do destinatario.

ETAPA 7 - Criacao do cliente TCP

Nesta etapa foi criada a camada de cliente TCP do projeto.

O objetivo desta etapa foi permitir que o usuario interaja com o servidor usando um cliente Java de console, sem depender de telnet ou de outros programas externos.

Arquivos criados:

1. br.edu.chat.client.ChatClient
   Classe responsavel por abrir a conexao TCP com o servidor.
   Ela cria um Socket apontando para o host e a porta configurados.
   Depois de conectar, cria duas threads:
   - uma thread para enviar comandos digitados no teclado;
   - uma thread para receber mensagens enviadas pelo servidor.

2. br.edu.chat.client.ConsoleInputThread
   Thread responsavel por ler o teclado do usuario usando BufferedReader sobre System.in.
   Cada linha digitada e enviada ao servidor pelo socket.
   O comando saircliente encerra apenas o cliente local, fechando o socket.

3. br.edu.chat.client.ServerListenerThread
   Thread responsavel por escutar mensagens vindas do servidor.
   Ela fica lendo o InputStream do socket e imprime cada linha recebida no console.
   Isso permite que o cliente receba mensagens mesmo sem estar digitando ou pressionando Enter.

4. br.edu.chat.MainClient
   Classe principal para iniciar o cliente TCP.
   Por padrao, conecta em localhost na porta 6789.
   Tambem permite informar host e porta por argumentos.

Funcionamento do cliente:

1. O usuario inicia o servidor.
2. O usuario inicia um ou mais clientes.
3. Cada cliente abre uma conexao TCP com o servidor.
4. O cliente cria duas threads:
   - ConsoleInputThread para envio;
   - ServerListenerThread para recebimento.
5. O usuario digita comandos no console.
6. Os comandos sao enviados ao servidor.
7. As respostas e mensagens recebidas sao impressas no console.

Testes previstos:

- iniciar um cliente;
- verificar mensagem de boas-vindas;
- enviar comando ajuda;
- cadastrar usuario;
- fazer login;
- iniciar dois clientes ao mesmo tempo;
- enviar mensagem de um cliente para outro;
- receber mensagens sem precisar pressionar Enter;
- encerrar um cliente com saircliente;
- verificar que o servidor continua rodando.

Limitacao conhecida:

Em console Java puro, usando System.in, nao ha controle completo da linha que o usuario esta digitando.
Quando uma mensagem chega enquanto o usuario digita, o cliente imprime a mensagem recebida e reimprime o prompt >.
Isso permite continuar a interacao, mas nao preserva perfeitamente o texto que estava sendo digitado.
Uma implementacao mais fiel desse requisito poderia usar uma biblioteca como JLine, que permite controlar a linha atual do terminal.

ETAPA 8 - Integracao de cadastro, login, logout, status e recuperacao de senha

Nesta etapa foram validadas e ajustadas as primeiras funcionalidades reais integradas entre cliente e servidor.

Funcionalidades validadas:

1. Cadastro de usuario
   O comando cadastro;Nome Completo;login;email;senha e interpretado pelo CommandParser e processado pelo CommandProcessor.
   O cadastro e encaminhado para UserService.register, que valida os campos, verifica login e email duplicados e chama UserRepository.create.
   A senha e salva no banco usando hash, por meio de PasswordUtils.

2. Login
   O comando login;login;senha e interpretado pelo CommandParser e processado pelo CommandProcessor.
   O UserService busca o usuario pelo login e valida a senha usando PasswordUtils.verify.
   Quando o login e valido, o status do usuario e alterado para ONLINE, o ClientHandler guarda o usuario logado e o ClientSessionManager registra o usuario como online.
   Depois disso, as mensagens pendentes sao entregues.

3. Logout
   O comando logout remove o usuario do ClientSessionManager, atualiza o status no banco para OFFLINE e limpa os dados de usuario logado no ClientHandler.
   Depois do logout, comandos protegidos voltam a exigir login.

4. Mudanca de status
   O comando status;ONLINE altera o status no banco para ONLINE, registra o usuario no ClientSessionManager e entrega mensagens pendentes.
   Os comandos status;OFFLINE, status;OCUPADO e status;AUSENTE atualizam o status no banco e removem o usuario do ClientSessionManager.
   Assim, o usuario pode continuar conectado ao socket, mas nao recebe mensagens imediatas enquanto nao estiver ONLINE.

5. Recuperacao de senha
   Como as senhas sao salvas com hash, nao e possivel exibir a senha original do usuario.
   Por isso, o comando recuperarsenha;email foi ajustado para gerar uma senha temporaria.
   A nova senha temporaria e mostrada ao usuario, o hash dela e salvo no banco e a senha antiga deixa de funcionar.

Correcoes realizadas nesta etapa:

1. Foi adicionada verificacao para impedir login simultaneo do mesmo usuario em dois clientes diferentes.
   Caso o usuario ja esteja conectado, o servidor retorna erro informando que ele ja esta logado em outro cliente.

2. Foi ajustada a recuperacao de senha para realmente alterar a senha do usuario.
   Antes, o sistema apenas retornava uma mensagem dizendo que a senha havia sido enviada.
   Agora, uma senha temporaria e gerada e gravada no banco.

3. Opcionalmente, foi adicionada validacao amigavel para nome completo duplicado, pois a tabela users possui restricao UNIQUE em full_name.

ETAPA 10 - Mensagem direta com aceite previo

Nesta etapa foi validado e corrigido o fluxo de mensagens diretas entre usuarios.

Comando principal:

@usuario: mensagem

Fluxo implementado:

1. Primeiro envio sem permissao previa
   Quando um usuario envia uma mensagem direta para outro usuario pela primeira vez, o sistema verifica:
   - se o remetente esta logado;
   - se o destinatario existe;
   - se nao existe bloqueio entre eles;
   - se ja existe permissao de conversa direta.

   Se nao existir permissao, o sistema:
   - salva a mensagem inicial na tabela messages;
   - cria um pedido pendente na tabela pending_requests;
   - associa o pedido pendente a mensagem inicial por meio da coluna pending_message_id;
   - envia o pedido de aceite ao destinatario, se ele estiver online;
   - responde ao remetente que a mensagem sera enviada apos aceite.

2. Aceite do pedido
   Quando o destinatario digita aceitar id, o sistema:
   - marca o pedido como aceito;
   - grava a permissao de conversa direta em direct_permissions;
   - notifica o remetente;
   - busca a mensagem inicial associada ao pedido;
   - entrega a mensagem inicial imediatamente se o destinatario estiver online;
   - ou salva a entrega como pendente se o destinatario nao estiver online.

3. Recusa do pedido
   Quando o destinatario digita recusar id, o sistema:
   - marca o pedido como recusado;
   - nao entrega a mensagem inicial;
   - notifica o remetente que o pedido foi recusado e que a mensagem inicial nao foi entregue.

4. Envios seguintes
   Depois que a permissao direta existe, novas mensagens de A para B sao enviadas sem novo pedido de aceite.
   Se o destinatario estiver online, a mensagem e entregue imediatamente.
   Se o destinatario estiver offline, ocupado ou ausente, a mensagem e salva como pendente e sera entregue quando ele voltar a ficar online.

Alteracoes realizadas:

1. Foi adicionada a coluna pending_message_id na tabela pending_requests.
   Essa coluna aponta para a primeira mensagem enviada antes do aceite.

2. A classe PendingRequest foi atualizada para armazenar pendingMessageId.

3. O RequestRepository foi atualizado para salvar e consultar pending_message_id.

4. O MessageRepository recebeu o metodo findById, usado para buscar a mensagem inicial quando o pedido e aceito.

5. O MessageService.sendDirect foi ajustado:
   - antes, a primeira mensagem sem permissao apenas criava o pedido de aceite;
   - agora, a primeira mensagem e salva e associada ao pedido pendente.

6. O RequestService.accept foi ajustado:
   - ao aceitar um pedido de mensagem direta, a permissao e concedida;
   - a primeira mensagem e entregue imediatamente ou salva como pendente.

7. O RequestService.refuse foi ajustado:
   - ao recusar um pedido de mensagem direta, o remetente e avisado que a mensagem inicial nao foi entregue.

Testes previstos:

- A envia mensagem direta para B pela primeira vez;
- B recebe pedido de aceite;
- a mensagem inicial nao aparece para B antes do aceite;
- B recusa e A e avisado da recusa;
- A tenta novamente e novo pedido e criado;
- B aceita;
- a mensagem inicial e entregue apos o aceite;
- A envia nova mensagem depois do aceite;
- B recebe a nova mensagem sem novo pedido;
- B fica offline;
- A envia mensagem para B;
- a mensagem fica pendente;
- B faz login;
- B recebe a mensagem pendente.

CORRECAO - Consistencia de logout, desconexao e usuarios online

Durante os testes com login, logout, Ctrl+C e saircliente, foi identificado que um usuario poderia continuar aparecendo em listausuarios mesmo depois de o cliente ter sido encerrado.

Causa identificada:

O comando listausuarios consulta os usuarios com status ONLINE no banco de dados.
Em alguns cenarios de encerramento abrupto, como Ctrl+C, fechamento do cliente ou queda de conexao, o banco poderia continuar com o usuario marcado como ONLINE se a limpeza da sessao nao fosse concluida corretamente.

Correcoes aplicadas:

1. ClientSessionManager
   Foi criada a remocao segura por handler.
   O metodo removeIfSameHandler remove o usuario da sessao somente se a sessao atual ainda pertencer ao mesmo ClientHandler.
   Isso evita que uma conexao antiga remova por engano uma sessao mais nova do mesmo usuario.

2. ClientHandler
   O metodo disconnect passou a ser idempotente, garantindo que a limpeza da sessao rode apenas uma vez.
   Quando o socket fecha ou ocorre erro de envio, o ClientHandler chama a rotina de desconexao.
   Essa rotina remove o usuario do ClientSessionManager, atualiza o status no banco para OFFLINE e limpa os dados do usuario logado.

3. ConsoleInputThread
   O comando saircliente passou a enviar logout ao servidor antes de fechar o socket.
   Assim, o encerramento voluntario do cliente tambem atualiza o usuario para OFFLINE.

4. ChatClient
   Foi adicionado um shutdown hook para fechar o socket em caso de Ctrl+C ou encerramento do processo do cliente.
   Isso aumenta a chance de o servidor detectar a desconexao e marcar o usuario como OFFLINE.

5. ChatServer
   Ao iniciar o servidor, todos os usuarios sao marcados como OFFLINE.
   Isso evita que usuarios fiquem presos como ONLINE apos queda ou reinicio abrupto do servidor.

Testes recomendados:

- Fazer login e depois logout; o usuario nao deve aparecer em listausuarios.
- Fazer login e depois saircliente; o usuario nao deve aparecer em listausuarios.
- Fazer login e fechar o cliente com Ctrl+C; o usuario nao deve aparecer em listausuarios apos a desconexao ser detectada.
- Reiniciar o servidor; usuarios de sessoes antigas nao devem aparecer como online.
- Alterar status para OCUPADO, AUSENTE ou OFFLINE; o usuario nao deve aparecer em listausuarios.
- Alterar status para ONLINE; o usuario deve voltar a aparecer em listausuarios.

ETAPA 12 - Adicionar usuario ao grupo

Nesta etapa foi validado e ajustado o fluxo de convite de novos membros para um grupo existente.

Comandos:

inserir &nomegrupo@usuario
inserir &nomegrupo@usuario1,@usuario2

Fluxo implementado:

1. O CommandParser interpreta o comando como INSERIR_GRUPO, extraindo o nome do grupo e a lista de usuarios convidados. Multiplos usuarios podem ser passados separados por virgula.
2. O CommandProcessor verifica se o usuario esta logado e, em seguida, repassa cada convidado para GroupService.insertMember.
3. GroupService.insertMember executa, para cada usuario convidado:
   - verifica se o grupo existe;
   - verifica se o remetente do convite e administrador do grupo;
   - verifica se o convidado existe na tabela users;
   - verifica se o convidado ja e membro do grupo;
   - verifica se ja existe um convite pendente do mesmo administrador para o mesmo convidado, evitando convites duplicados.
4. Caso o convite seja valido, e criado um registro em pending_requests com request_type = INVITE e status = PENDING.
5. Se o convidado estiver online, o servidor envia imediatamente uma mensagem no formato:
   [CONVITE id] usuario te convidou para o grupo 'nomegrupo'. Use 'aceitar id' ou 'recusar id'.
   Caso esteja offline, o convite fica salvo em pending_requests e e reenviado quando o usuario voltar a ficar online.

Aceite e recusa:

- O comando aceitar id e tratado por RequestService.accept. Para um convite, o sistema:
  - marca o pedido como ACCEPTED;
  - adiciona o usuario na tabela group_members;
  - avisa o remetente do convite, se ele estiver online;
  - avisa os demais membros do grupo com a mensagem "usuario entrou no grupo 'nomegrupo'.".
- O comando recusar id marca o pedido como REFUSED e avisa o remetente do convite que o pedido foi recusado.

Ajustes realizados nesta etapa:

1. Notificacao do grupo ao aceitar convite
   O metodo acceptInvite no RequestService passou a chamar notifyGroupMembers para que todos os outros membros do grupo recebam a mensagem [SISTEMA] usuario entrou no grupo 'nomegrupo'.

2. Entrega de convites pendentes ao login
   Foi criado o metodo deliverPendingForUser em RequestService. Quando o usuario faz login ou muda status para ONLINE, o servidor:
   - busca pedidos pendentes onde target_user_id = id do usuario;
   - reenvia as mensagens de convite, pedido de mensagem direta, pedido de entrada em grupo e convite de administracao;
   - usa os mesmos formatos enviados na hora original.
   Esse metodo e chamado por CommandProcessor logo apos a entrega de mensagens pendentes.

Pontos validados:

- Administrador convida usuario existente para o grupo.
- Convidado online recebe o convite imediatamente.
- Convidado offline recebe o convite ao fazer login.
- Convidado aceita e passa a aparecer em listargrupo &nomegrupo.
- Grupo recebe aviso de entrada do novo membro.
- Convidado recusa e nao entra no grupo.
- Convite duplicado para o mesmo usuario nao e criado.
- Usuario que ja e membro nao recebe convite e o sistema informa.
- Usuario nao administrador nao consegue inserir ninguem no grupo e recebe erro.
- O comando inserir aceita varios usuarios separados por virgula, criando um convite para cada um.


ETAPA 13 - Enviar mensagem para grupo

Nesta etapa foi validado o envio de mensagens publicas dentro de um grupo.

Comando:

&nomegrupo: mensagem

Fluxo implementado:

1. O CommandParser interpreta a linha como GROUP_MESSAGE, extraindo o nome do grupo e o conteudo apos os dois pontos.
2. O CommandProcessor encaminha para MessageService.sendGroup.
3. MessageService.sendGroup executa as verificacoes:
   - usuario remetente esta logado e em estado que permite envio;
   - o grupo existe na tabela groups_chat;
   - o remetente e membro do grupo.
4. A mensagem e salva em messages com message_type = GROUP e group_id apontando para o grupo.
5. Para cada membro do grupo, exceto o remetente, o sistema:
   - verifica se existe bloqueio entre o remetente e o membro; se houver, a entrega e ignorada e contada como "bloqueado";
   - se o membro estiver online, entrega imediatamente com o formato [nomegrupo] usuario (data): mensagem e marca a entrega como concluida em message_deliveries;
   - se o membro estiver offline, ausente ou ocupado, a entrega fica como pendente para ser entregue depois.
6. O servidor responde ao remetente com um resumo: quantos receberam agora, quantos receberao depois e quantos foram bloqueados ou nao receberam.

Pontos validados:

- Membro envia mensagem ao grupo e todos os outros membros online recebem.
- Remetente recebe confirmacao com a contagem de entregas.
- Membro offline nao recebe na hora, mas recebe depois ao ficar online.
- Bloqueio entre remetente e membro impede a entrega; demais membros nao bloqueados continuam recebendo.
- Usuario que nao e membro do grupo nao consegue enviar para o grupo.


ETAPA 14 - Mensagem direta dentro do grupo

Nesta etapa foi validado o envio de mensagens privadas para um ou mais membros do grupo.

Comandos:

&nomegrupo@usuario: mensagem
&nomegrupo@usuario1,@usuario2: mensagem

Fluxo implementado:

1. O CommandParser interpreta a linha como GROUP_DIRECT_MESSAGE, extraindo o nome do grupo, a lista de destinatarios e o conteudo da mensagem.
2. O CommandProcessor encaminha para MessageService.sendGroupDirect.
3. MessageService.sendGroupDirect executa as verificacoes:
   - usuario remetente esta logado e em estado que permite envio;
   - o grupo existe;
   - o remetente e membro do grupo.
4. Para cada destinatario informado, o sistema:
   - busca o usuario pelo login;
   - verifica se o destinatario e membro do grupo e nao e o proprio remetente;

- Destinatario offline recebe a mensagem quando ficar online.
- O formato [nomegrupo | privada] permite ao destinatario diferenciar de mensagens publicas do grupo.


ETAPA 15 - Solicitar entrada em grupo existente

Nesta etapa foi validado e ajustado o fluxo de solicitacao de entrada em um grupo, no qual todos os membros atuais precisam aprovar a entrada de um novo participante.

Comando:

entrar &nomegrupo

Regra do enunciado:

Para um usuario entrar em um grupo existente, todos os membros atuais devem aceitar. Se qualquer membro recusar, a entrada e negada e as demais pendencias sao canceladas.

Fluxo implementado:

1. O CommandParser interpreta a linha como ENTRAR_GRUPO, extraindo o nome do grupo.
2. O CommandProcessor encaminha para GroupService.requestJoin.
3. GroupService.requestJoin executa as verificacoes:
   - o grupo existe;
   - o usuario ainda nao e membro do grupo;
   - nao existe pedido de entrada pendente do mesmo usuario para o mesmo grupo.
4. O sistema busca todos os membros atuais do grupo e cria, para cada um, um registro em pending_requests com request_type = GROUP_JOIN, requester_user_id = id do solicitante, target_user_id = id do membro e group_id do grupo.
5. Para cada membro online, o servidor envia imediatamente:
   [SOLICITACAO id] usuario quer entrar no grupo 'nomegrupo'. Use 'aceitar id' ou 'recusar id'.
   Membros offline recebem a solicitacao ao ficar online, atraves de deliverPendingForUser.
6. O servidor responde ao solicitante informando que o pedido foi enviado a todos os membros atuais.

Aceite:

- Quando um membro digita aceitar id, RequestService.accept executa acceptGroupJoin:
  - marca o pedido daquele membro como ACCEPTED;
  - consulta quantas aprovacoes ainda estao pendentes para o mesmo solicitante e grupo (countPendingGroupJoinApprovals);
  - se ainda restarem aprovacoes pendentes, responde informando quantas faltam;
  - se nao restarem, adiciona o solicitante em group_members, avisa o solicitante de que o pedido foi aceito por todos e avisa o grupo com a mensagem [SISTEMA] usuario entrou no grupo 'nomegrupo'.

Recusa:

- Quando um membro digita recusar id, RequestService.refuse executa:
  - marca o pedido daquele membro como REFUSED;
  - chama RequestRepository.refusePendingGroupJoinApprovals para marcar como REFUSED todas as demais solicitacoes pendentes do mesmo usuario para o mesmo grupo;
  - avisa o solicitante de que seu pedido foi recusado.
  Assim, basta uma recusa para encerrar todo o fluxo.

Ajustes realizados nesta etapa:

1. Notificacao do grupo ao aceite final
   O metodo acceptGroupJoin no RequestService passou a chamar notifyGroupMembers apos confirmar que todas as aprovacoes ja foram coletadas. Todos os membros online recebem a mensagem de entrada do novo participante.

2. Notificacao em segundo plano ao logar
   Como cada membro pode estar offline no momento da solicitacao, o metodo deliverPendingForUser do RequestService reenvia o pedido de entrada quando o membro entra novamente. Assim, o requisito de mensagens offline tambem se aplica a esta etapa.

Pontos validados:

- Usuario solicita entrada e todos os membros recebem a solicitacao.
- Apenas com a aprovacao de todos o usuario entra no grupo.
- A entrada do usuario gera aviso para os demais membros.
- Se um membro recusar, o solicitante e avisado e nao entra no grupo.
- Solicitacoes restantes sao canceladas automaticamente apos a recusa.
- Usuario ja membro nao pode solicitar entrada novamente e recebe erro.
- Membro offline no momento do pedido recebe a solicitacao ao logar.
